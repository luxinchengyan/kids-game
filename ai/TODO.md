## 一、总体判断

- **产品文档与实现落差较大**：PRD 中的「童梦乐园」「AI 引擎」「家长中心」「多岛场景」等在现网主要是「单页任务流 + localStorage」，需明确 **MVP 边界** 并在文档或产品上对齐预期，避免对外演示时过度承诺。
- **技术栈与工程化偏轻**：无单元测试、入口文件重复、部分关键学习数据未接入知识图谱，适合快速验证，但距离「世界一流商业项目」的 **可测试性、可观测性、无障碍（读屏等）** 仍有系统差距。

## 三、P1：产品与设计

### 3.1 产品结构与信息架构

- **单一路径**：当前以「一轮约 6 题」为主流程，缺少 PRD 中的大厅、选岛、进度地图；儿童端 **菜单仍偏「控制台化」**（如英文标签 Companion、Commercial MVP）。应统一为 **幼儿向中文**，并完成真正的「一屏一事」导航。
- **家长价值闭环不完整**：有摘要与 JSON 导出，但缺少 **可控学习时长、每日目标、周报、多子女档案** 等家长决策刚需；与 PRD「家长中心」差距大。
- **奖励与成就**：成就体系已有数据层，但 **触达儿童的展示与仪式感**（全屏庆祝、图鉴）可加强，避免仅为后台计数。

### 3.2 视觉与交互（对照 DESIGN.md）

- **设计令牌未落地**：DESIGN 中的魔法彩虹色板与组件级规范（触控 ≥64dp、圆角体系）在 `styles.css` 中仅为另一套 token，**未建立单一设计系统**（易漂移）。
- **单一中文界面**：移除或废弃英文 locale；所有可见字符串走统一中文文案表，避免 `t()` 与硬编码混用长期分裂。
- **反馈时序**：项目规则要求「200ms 内视觉反馈」；需对按钮 `active`、加载、错题引导做 **统一动效与禁用连点** 审计（Framer Motion 已引入但使用面偏窄）。

### 3.3 性别、角色与默认绑定（产品定调）

- **仅男 / 女两档**：与小朋友在档案中登记的 **生理性别（男、女）** 一致；**不提供**「其他」「非二元」等选项，避免对幼儿及家长造成多余认知负担，界面与数据模型一并收敛。
- **默认绑定策略**：根据档案性别 **默认匹配** 男孩线 / 女孩线陪伴角色与主题皮肤（见下节「IP 物料」）；可在家长同意下允许 **在同性别线内切换皮肤或伙伴**，但不跨性别线混用主视觉（便于周边与品牌识别一致）。
- **错题路径**：如 `PinyinBattle` 答错仅文案提示，无 **非惩罚性脚手架**（再听一遍高亮、慢速播放、二选一简化）；与 DESIGN「容错设计」可再对齐。

### 3.4 男 / 女双线 IP 全套物料（周边预备）

为男孩、女孩分别建立 **可授权、可延展的原创角色与世界观**，并形成可交付资产清单，供 UI、宣发与 **实体周边** 打样：

| 类别 | 男孩线 | 女孩线 | 备注 |
|------|--------|--------|------|
| 角色设定 | 主陪伴角色名、性格、口头禅、色板 | 同上 | 需避开第三方 IP，自有商标注册路径单列 |
| 头像 / 立绘 | 多表情、多年龄感 Q 版 | 同上 | 矢量 + 位图导出规格统一 |
| 场景背景 | 大厅、学习岛、战斗底图等分层 PSD/FIG | 同上 | 动效层与静态层分离 |
| 声音 | 男声童声 / 青年男声配音线索、音效包 | 女声童声 / 青年女声配音线索、音效包 | 含按钮音、胜利短音、错误柔和音 |
| UI 组件皮肤 | 按钮、进度条、奖励框主题变体 | 同上 | 与设计 token 绑定，避免硬编码色值 |

**验收**：每条线具备 **风格指南一页纸**（字体、圆角、禁用色、安全区）+ **资源命名规范**，便于工厂与外包批量扩产。

## 四、P1：工程与质量

- **测试**：`package.json` 中 `test` 为占位；已有 Playwright e2e，但 **核心业务逻辑**（出题、`scheduleNextReview`、成就条件）无单元测试，回归风险高。
- **双入口维护**：同时存在 `main.tsx` 与 `main.jsx`，易造成混淆；应保留单一入口并统一 TypeScript 策略（要么全量 TS，要么明确仅入口为 TS）。
- **类型与安全**：大量业务在 `.jsx` 中，依赖运行时；长期建议 **关键模块 TS 化**（store、learningContent 生成器）。
- **性能与资源**：拼音音频依赖 `/audio/*.mp3`，需确认 **资源完整性、CDN 与失败降级**（已有 TTS fallback，但音质与教学一致性需产品确认）。

## 五、教育与内容（拼音 · 数学 · 英语 · 传统文化）

### 5.1 汉语拼音与语音

- **听辨题选项构造**：`buildPinyinBattle` 用 `[unit.content, ...confusionSet].slice(0,3)`，混淆项来自内容数据中的简写（如 `'p','d'`），与正式拼音教学中的 **音位对立组** 未必一致；建议按 **课标与常见偏误** 校验每一组选项是否可比、可听辨。
- **TTS 与专业录音**：`audio.js` 中间用单字映射做朗读，**多音字、轻声、儿化** 无法覆盖；商业级产品通常需 **真人录音 + 口型/拼音卡动画** 配套。
- **整体认读音节与拼写**：数据已覆盖整体认读；**拼读关卡**与 **字母分解** 的进阶关系需在任务序列中显式体现（脚手架），避免跳跃。

### 5.2 数学与英语（内容模块）

- **数学**：题目以数星星、简单加法为主，与 **数感、一一对应、模式** 等学前核心经验可更丰富；难度与 `ageGroup` **未强绑定**。
- **英语（作为学习内容，非界面语言）**：可保留「英语游乐园」等 **中文环境下的英语启蒙** 模块；界面仍为中文。需教研补充 **听力输入量、字母意识、Phonics** 与年龄的映射说明。

### 5.3 中国传统文化与故事线

- `storyData` 已含神话、古诗、成语等 **优质母题**，但 **未进入可玩任务流**（无 `buildStoryTask` 一类）；与 PRD「故事王国」差距明显。
- 部分拼音例词涉及 **宗教相关用语**（如 blend 条目中与「佛」相关的例词），在幼儿产品与公立园场景可能敏感，建议 **替换为中性、生活化例词**。
- 古诗/成语若上线，需配套 **适龄释义、正向价值观导读** 与家长提示，避免仅停留于标题罗列。

## 六、合规、隐私与商业化基础

- **儿童与个人信息**：上线账号与支付后，须符合 **《个人信息保护法》** 及 **未成年人网络保护** 相关要求：家长同意、隐私政策摘要、儿童专用规则、最小必要收集；若仅存 localStorage 的过渡期，也应在 UI 说明数据范围。
- **家长同意与广告**：若含 **分享、好友家庭绑定、对外展示学习数据** 等能力，须在隐私政策与首次使用时逐项说明并取得家长同意；当前无 **使用时长锁定** 与 **休息提醒**（DESIGN 已写，代码侧未落实）。
- **内容审核**：UGC 或 AI 生成若引入，需审核 pipeline；当前纯静态数据风险较低。

### 6.1 商业化与付费方案（产品级）

面向中国大陆，建议采用 **手机号 + 微信** 双主通道，付费颗粒度覆盖试用与长短周期：

| 模块 | 方案要点 |
|------|-----------|
| **登录 / 注册** | **手机号 + 短信验证码**（必选基础能力）；**微信扫码登录**（家长常用设备，降低流失）；可选 **本机号码一键登录**（运营商 SDK，视平台而定）。儿童学习端不单独注册账号，由家长账号绑定子档案（男 / 女）。 |
| **试用** | **首周免费 / 7 天全功能试用**（从首次登录或首次开通算起，产品定一条清晰规则）；试用结束前推送续费说明（站内 + 模板消息，若接微信）。 |
| **付费类型** | **按次**（例如单次解锁某岛或单次 AI 互动包）；**按周**；**按月**；**按年**（年付折扣）。具体定价与权益档位由财务与市场定，技术侧预留 **订单类型、到期时间、自动续费开关** 字段。 |
| **支付渠道** | 微信内：**微信支付**；App / H5：**微信支付 + 支付宝**（以实际上架平台为准）。 |
| **订阅管理** | 家长中心内：**当前权益、到期时间、发票 / 凭证入口（如需）**、**关闭自动续费**；符合监管展示要求。 |

**工程侧后续依赖**：服务端用户体系、订单与回调验签、收据对账；本仓库前端为 MVP 时可用 **mock + 接口契约** 先行。

### 6.2 微信生态：好友触达、分享与家长交流（增长与激励）

**产品目标**：方便 **家长之间** 交流孩子的学习进度、经验与打卡；通过 **分享与同伴激励** 提升孩子坚持学习的动力（正向反馈、荣誉感，而非攀比压力）。

| 能力 | 产品用途 | 说明与实现注意 |
|------|-----------|-----------------|
| **分享** | 将「今日学习小结」「连续打卡天数」「解锁成就」「薄弱点进步」生成 **图文卡片 / 海报**，一键分享到 **微信会话或朋友圈**（以平台开放能力为准） | H5 常用 **微信 JS-SDK**（`updateAppMessageShareData`、`updateTimelineShareData` 等，接口名以微信官方文档为准）；**小程序**内可走 `button open-type="share"`。分享文案与缩略图需过审：**不出现具体幼儿姓名默认可勾选脱敏**（仅显示昵称或「我家宝贝」）。 |
| **邀请家长 / 同班交流** | 家长生成 **带渠道参数的邀请链接或小程序码**，其他家长扫码进入后绑定「同学 / 好友家庭」关系，用于 **进度对比（可选关闭）**、**组队打卡**、**私信或群聊入口**（可跳转企业微信 / 微信群二维码由运营配置） | 网页端 **无法** 像系统通讯录那样任意「读取全部微信好友列表」，需采用 **分享即邀请、扫码绑定、搜索邀请码** 等产品形态实现「拉好友」。若需更强社交链，优先 **微信小程序 + 服务端关系链**。 |
| **学习进度同步给好友家长** | 在 **家长授权** 前提下，向已绑定的好友家庭推送「本周报告摘要」或允许对方查看 **脱敏后的学习时长 / 星星数区间** | 默认 **关闭**，需逐项授权；支持 **随时解绑** 与 **删除关系**。 |
| **激励小朋友** | 分享图含 **孩子自选虚拟形象、成就贴纸、本周金句**；可选「好友家孩子也完成了」类 **匿名化同伴提示**（双方家长均同意才开启） | 避免公开排名与真实姓名；文案由教研审过，符合幼儿心理。 |

**合规要点**：涉及 **未成年人信息与社交关系**，须在隐私政策中单独说明目的、范围与保存期限；**不向儿童默认开启**任何对外展示或加好友类功能，入口放在 **家长中心** 且需验证家长身份（与 §6.1 登录体系一致）。

**工程侧**：微信各接口以 **微信公众平台 / 开放平台** 最新文档为准；需 **已备案域名、HTTPS、JS 安全域名配置**；小程序与 H5 若并存，需统一 **UnionID / 业务用户 ID** 映射。

### 6.3 数据统计、埋点与运营看板

**目标**：用可解释的数据驱动 **产品迭代、投放 ROI、客服与教研**，同时满足 **最小必要、家长知情** 原则（与隐私政策、未成年人条款一致）。

| 层级 | 建议内容 |
|------|-----------|
| **采集方式** | **前端埋点**（页面/按钮/任务流漏斗）+ **服务端业务日志**（登录、订单、权益变更，以服务端为准防篡改）+ **错误监控**（JS 报错、接口失败率、白屏率）。 |
| **国内常用能力选型** | 需法务评估 SDK 清单：**移动统计 / 分析类**（如友盟、神策、TalkingData、腾讯移动分析等，以最新合规公示为准）；**小程序**使用微信官方数据分析 + 自选分析平台；**Web/H5** 注意 **Cookie/本地标识** 与备案要求。具体厂商以采购与合规结论为准，本文件不绑定单一供应商。 |
| **事件模型（示例）** | 统一 **事件名中英字典**（如 `task_start`、`task_complete`、`pay_success`、`share_click`、`trial_end`）；关键属性：`channel`（渠道码）、`sku`（付费档）、`child_profile_segment`（仅存年龄段枚举，不存真实姓名）、`session_id`。 |
| **漏斗与留存** | **激活 → 建档 → 首局完成 → 次日回访 → 试用结束 → 付费转化**；按渠道、性别线、年龄段拆分；**功能留存**（拼音岛 / 数学 / 英语）单独看板。 |
| **投放与归因** | H5/小程序链接带 **utm 或自研 channel 参数**；应用商店用 **分包或渠道包**（若上 App）；与 §6.2 邀请码 **合并设计**，避免多套 ID。 |
| **工程实现要点** | 封装 `track(event, props)`，**开发 / 预发可关闭或打桩**；生产环境 **采样率**（如错误 100%、行为事件可配置）以控成本；**禁止**在埋点中上传幼儿真实姓名、精确住址、学校全名等。 |
| **看板受众** | 产品 / 运营看 **漏斗与功能使用**；研发看 **稳定性与性能**；教研看 **题型正确率、耗时分布**（需脱敏聚合）。 |

## 十一、成功指标建议（产品可量化）

在未接入数据分析服务前，可用 **家长侧可观察指标 + 工程侧代理指标** 定义「这一轮迭代是否变好」：

- **完成率**：进入任务流的用户（或测试账号）完成 **≥80%** 当日题目的比例（可用 e2e 或手工基准）。
- **误触与挫败**：同一题连续错误 **≥3 次** 的会话占比（需埋点或日志后才可算；前期可用用户测试样本代替）。
- **时长可信度**：家长摘要中的练习时长与真实会话起止误差在可接受范围（修复 `responseTime`、明确 `sessionStartedAt` 语义后）。
- **内容安全**：静态例词、角色名 **零未授权 IP、零不适宜幼儿场景的敏感词**（人工抽检清单）。
- **分享与拉新（上线 §6.2 后）**：分享按钮点击率、邀请链接 **有效激活率**（受邀家长完成注册 / 绑定子档案）、家长对「好友进度」功能的 **开启率与投诉率**。
- **数据与投放（上线 §6.3、§6.4 后）**：各 **渠道 ROI**（激活成本、付费转化率、LTV 粗算）、**关键漏斗每步流失率**、**JS 错误率 / 接口错误率**、首屏 **LCP**（与统计脚本加载策略相关）。

## 十三、待产品拍板的问题（开放项）

以下不强行给技术方案，需产品 / 教研共同决定后，再写入 PRD 子章节：

1. **品牌唯一名称** 与 **是否保留「岛 / 乐园」子品牌** 的层级关系。  
2. **英语内容模块**：启蒙单词为主，还是自然拼读为主，与拼音课是否 **同学期混排**（界面始终中文）。  
3. **传统文化**：神话/古诗在游戏中是 **轻量彩蛋** 还是 **独立关卡线**，与识字量前提。  
4. **家长端**：是否需要 **独立入口（PIN / 长按）** 防止幼儿误触设置与导出。  
5. **商业化细节**：各档位价格、年付折扣、按次权益边界、试用是否限设备、**自动续费**提前扣款提示文案（合规）。  
6. **男 / 女双线**：两条 IP 世界观的命名与主色是否在宣发上 **完全分线** 或 **共享主品牌 + 子系列**。  
7. **§6.2 社交深度**：是否需要 **小程序内闭环**（关系链 + 消息），还是 **H5 分享 + 跳转微信群/企微** 即满足 MVP。  
8. **同伴激励**：是否允许「好友家孩子匿名进度」类提示，以及 **完全关闭攀比向文案** 的红线表述。  
9. **数据分析**：优先自建行为库还是采购 SaaS；**年度数据预算** 与看板人数。  
10. **首年推广**：线下园所 vs 线上投放的 **预算比例** 与核心 KPI（激活成本、付费转化率）。

## 十四、仍可改进与查漏补缺（归纳）

除前文各节已列事项外，下列 **跨模块** 问题可持续对照、避免遗漏：

- **观测性**：上线后若无 §6.3，则 **无法** 科学评估功能改版与投放效果；建议与 **首版付费** 同步最小埋点包（激活、首局完成、付费成功、分享点击）。  
- **渠道一致性**：应用商店描述、落地页、公众号菜单中的 **产品名、试用规则、价格** 须与 App / 小程序内 **完全一致**，避免虚假宣传客诉。  
- **客服与危机预案**：**未成年人误充值、家长要求删号导出数据** 等流程需书面 SOP，与 §6.1、合规条款对齐。  
- **性能与成本**：投放带来流量峰值时，**CDN、接口限流、统计采样** 需提前压测，防止统计脚本拖慢首屏。  
- **教研闭环**：埋点中的 **题型维度** 应能回传到内容侧，驱动 **题目难度与混淆项** 迭代（与 §五联动）。  
- **竞品与差异化**：定期复盘 **同类启蒙 App** 的定价、功能、评价区痛点，更新本 TODO 与 PRD 的 **差异化一句**。
